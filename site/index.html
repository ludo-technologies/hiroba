    一Why

    バーチャルオフィスは、
    重くなりすぎた。

    ビデオ・画面共有・録画・連携——便利になるほど、常駐させるには重いツールになっていく。 一日中つなぎっぱなしにする「在席ツール」に必要なのは、高度な会議機能ではなく、気配と軽さです。

    従来の多機能型ツール

    • 通信量（音声中心でも）約1GB/日
    • ビデオ・画面共有を多用すると3〜5GB/日
    • 常駐時のメモリ数百MB〜
    • 位置づけ会議ツールの代替

    Hiroba

    • 通信量音声のみ・低帯域
    • アイドル時CPUほぼ0%
    • 常駐時のメモリ数十MB台
    • 位置づけ常駐前提の在席ツール

    二Features

    Hiroba でできること。

    バーチャルオフィスに必要な最小限——在席・気配・声だけを備えています。 ビデオ会議・画面共有・録画は、軽さのためにあえて非搭載。会議はいまお使いのツールのままで。

    一

    アバターで出社する

    朝、フロアに自分のアバターを置く。それが出社。リモートでも「居場所」ができる。

    二

    誰がどこにいるか見える

    常設のメンバー一覧で、在席・離席・通話中・取り込み中が常にひと目で分かる。

    三

    フロアを歩き回れる

    2Dの見下ろしフロアを移動できる。「ちょっと席まで行って話しかける」が画面の中でできる。

    四

    近づけば、声がつながる

    チームの部屋は入るだけでグループ通話。離れた相手にもワンクリックで「呼びかけ」。

    五

    つけっぱなしでも軽い

    アイドル時CPUほぼ0%・メモリ数十MB。8時間常駐させても、PCの邪魔をしない。

    三How

    オフィスの中身は、ロビーとチームの部屋。

    組織にひとつの仮想フロア。中に全員のロビーと、チームごとの小部屋があるだけ。迷う構造はありません。

    ロビー — 組織共通のオープンフロア

    近づくと声がフェードイン

    チーム A（〜5人）

    実質グループ通話

    チーム B（〜5人）

    タブで出入り

    呼びかけ＝スペースを跨ぐ即時1:1。相手がどこに居ても、ワンクリックで声が届く

    ロビーは「気配」の場所

    近接音声のオープンフロア。歩み寄ると声がフェードインする。雑談やすれ違いが生まれる場所です。

    チームは「小さな声の輪」

    5人前後のチームスペースは全員が常時つながる実質グループ通話。タブひとつで輪に出入りできます。

    呼びかけは「肩を叩く」

    一覧から相手を選んで呼びかけると、スペースを跨いで即時1:1の声がつながる。取り込み中（DND）はきちんと遮断。ワンクリックで切断。

    声はサーバーを通らない

    音声はWebRTCのP2Pメッシュで直接つながり、サーバーは位置と在席の中継だけ。だから軽く、運用も単純です。

    四Light

    なぜ、ここまで軽くするのか。

    在席ツールは8時間つけっぱなしが前提。動作が重ければ各自が閉じてしまい、 誰もいないオフィスだけが残ります。軽さは売り文句ではなく、 バーチャルオフィスが“オフィスとして機能する”ための前提条件です。

    0% アイドル時CPU 移動・発話なしのとき、ほぼゼロ
    数十MB 常駐メモリ ブラウザSPA比で一桁以上の削減
    <300ms 音声遅延 P2P直結。体感で違和感のない範囲
    数秒で起動 起動時間 Tauri製。Electronより小さく軽く

    数値は設計目標値（要件定義 NFR-01〜05）。実測の継続検証とともに更新します。

    五Open Source

    オープンソースです。

    サーバーもクライアントも、すべて Apache-2.0 で公開。 中身を読んで確かめられて、自分のサーバーでも動かせます。

    terminal
    $ git clone https://github.com/ludo-technologies/hiroba
$ cd hiroba/server && cargo run
✓ hiroba-server listening on 0.0.0.0:8787
# 単一バイナリ・DB不要。これだけでフロアが立ち上がります
    • 単一バイナリ・DBレスRust製サーバーは外部依存なし。社内ネットワークでも数分で self-host できます。
    • 音声はサーバーを通らないサーバーが扱うのは位置と在席の制御データだけ。音声はWebRTCのP2Pで直接つながります。
    • 仕様まで公開ワイヤープロトコル（PROTOCOL.md）や要件・実装状況のドキュメントも、すべてリポジトリにあります。
    • self-host は一級市民ホスト型と同じコードベース。無料・席数無制限で、機能の出し惜しみはありません。

    六Start

    始め方は、ふたつ。

    運用おまかせのホスト型と、自前で立てるOSS。同じコードベースから生まれる二つの配布です。

    Hosted

    ホスト型

    インフラはこちらで運用。ログインして組織をつくり、招待リンクを配るだけ。非技術チームでもすぐ使えます。

    • OAuthログイン（Google / GitHub など）
    • 組織の作成・招待リンク・メンバー管理
    • TURN込みで「ほぼ必ず繋がる」接続性
    • シート課金（料金ページ参照）
    早期アクセスに登録→

    Open Source

    OSS self-host

    単一バイナリのサーバーを自前で起動。DBレス・ゲスト入室で、社内ネットワークでも数分で動きます。

    • Apache-2.0（特許条項つき）
    • 単一バイナリ・外部依存なし（DB不要）
    • ゲスト/簡易入室、OAuthは任意で有効化
    • セットアップ手順・プロトコル仕様を公開
